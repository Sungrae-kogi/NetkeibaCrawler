# Netkeiba Master Crawler Pipeline

## 프로젝트 개요 (Overview)
본 프로젝트는 일본 경마 플랫폼 'Netkeiba(넷케이바)'의 방대한 데이터를 체계적으로 수집, 가공하여 csv 파일로 적재하는 완전 자동화 파이프라인입니다. 
초기에는 단일 웹 페이지의 경기 결과를 수집하는 단순한 스크립트로 시작했으나, 현재는 과거 경기 결과, 미래 출전표, 날씨, 말/기수/조교사의 상세 프로필, 그리고 실시간 경기 변동 정보까지 전방위적으로 데이터를 추적 및 관리하는 거대한 시스템으로 발전했습니다.

---

## 시스템 아키텍처 및 주요 구성 (Architecture)
시스템은 `all.py` 마스터 오케스트레이터를 중심으로 여러 독립적인 모듈이 유기적으로 연결되어 동작합니다.

1. **Master Orchestrator (`all.py`)**
   - 전체 파이프라인을 제어하는 메인 컨트롤 타워입니다.
   - 대화형 CLI를 통해 과거 경기 데이터 수집, 미래 출전표 수집, 실시간 모니터링 모드를 분기하여 실행합니다.
   - 정해진 룰에 따라 대상 경주를 자동으로 탐색(Discovery)하여 수동 URL 입력의 번거로움을 제거했습니다.

2. **인증 및 세션 관리 자동화 (`netkeiba_auth.py`)**
   - Playwright를 활용하여 Netkeiba 프리미엄 계정 자동 로그인 로직을 구현했습니다.
   - 세션 정보를 `storage_state.json`으로 유지하여 매 실행마다 발생하는 로그인 부하와 계정 차단 위험을 최소화했습니다.

3. **메인 데이터 수집 파트 (`WebCrawler`, `WeatherCrawler`)**
   - 경주의 기본 정보, 순위, 그리고 출전표(Entry Sheet) 데이터를 1차적으로 파싱합니다.
   - 난독화된 JS 기반의 실시간 날씨 및 마장 상태 예보 데이터를 디코딩하는 별도의 날씨 크롤러가 통합되어 있습니다.
   - 파싱 과정에서 발견된 고유 식별번호(말, 기수, 조교사 ID)를 추출하여 하위 크롤러들에게 자동으로 분배(Dispatch)합니다.

4. **엔티티 프로필 딥 다이브 (`HR/JK/TR Crawler`)**
   - 분배받은 ID를 기반으로 말(HRNO), 기수(JKNO), 조교사(TRNO)의 상세 프로필과 과거 누적 성적을 심층적으로 수집합니다.
   - 터미널에 진행률(`[i/n] TRNO=...`)을 로깅하여 작업 가시성을 높였습니다.

5. **실시간 모니터링 (`InformationCrawler`)**
   - 백그라운드에서 일정 간격으로 넷케이바 홈페이지의 Information 탭을 모니터링합니다.
   - 출주 취소, 기수 변경 등 경마에 직접적인 영향을 미치는 실시간 공지사항을 정규화하여 추출합니다.

6. **데이터베이스 적재 (Database Integration)**
   - 수집 완료된 파편화된 CSV 파일들을 MariaDB로 전송하여 자동 병합 및 갱신(Upsert)함으로써 데이터의 무결성을 보장하고 분석 용이성을 높였습니다.

## 문제 해결 사례 및 로직 개선 (Troubleshooting)

### 🚨 실패 사례: 복잡한 자가 복구(Auto-Healing) 로직이 초래한 데이터 결함
프로젝트 중반, 하위 크롤러(HR/JK/TR) 단계에서 네트워크 단절에 대비하기 위해 **'이어받기(Merge-and-Skip)'** 로직을 도입했습니다. 수집된 파일을 매번 읽어와서 누락된 타겟만 덧붙이는 방식이었습니다.
**[문제 발생]** 
네트워크 불안정으로 크롤러가 비정상 종료되었다가 재시작될 때, 기존 CSV 파일을 병합하는 과정에서 들여쓰기(Indentation) 오류와 열 밀림(Column Shift) 현상이 심각하게 발생했습니다. 부분적인 업데이트가 파일 전체의 구조를 망가뜨려 파이프라인이 멈추는 병목 원인이 되었습니다.

**[해결 및 아키텍처 개편]**
파일 I/O 단계를 극도로 단순화하고 구조를 전면 개편하여 이 문제를 해결했습니다.
1. **Overwrite-on-run 모델 전환**: 기존의 복잡한 이어받기 병합 코드를 과감히 삭제하고, 실행 시마다 깔끔하게 전체를 덮어쓰는(Overwrite) 단순한 구조로 회귀했습니다.
2. **강력한 Network Retry 적용**: 대신 통신 세션 자체에 강력한 재시도(Retry) 로직과 예외 처리(Exception Handling)를 부여하여 중간에 크롤링이 끊기는 원인 자체를 차단했습니다.
3. **결과**: `all.py`에서 파이프라인을 통제하기가 훨씬 수월해졌고, 파일 구조가 깨지는 버그가 완벽하게 사라져 24/7 구동 가능한 안정성을 확보했습니다.

## 향후 개선 방향 (Future Improvements)
프로젝트 코드를 전체적으로 스캔해 본 결과, 파이프라인을 한 단계 더 발전시키기 위해 다음과 같은 방향의 개선을 제안합니다.

* **DB Direct Streaming 도입**: 현재는 `nodata/` 와 `data/` 디렉토리에 중간 단계의 CSV 파일을 계속 생성하며 읽고 쓰고 있습니다. 규모가 커질수록 디스크 I/O가 늘어나므로, 데이터를 수집하는 즉시 메모리단에서 MariaDB로 직행(Streaming)하는 아키텍처로 변환을 고려해볼 수 있습니다.
* **비동기(Async) 크롤링 확장**: 하위 크롤러들이 순차적으로(Synchronous) 리퀘스트를 보내고 있습니다. 넷케이바 서버에 무리를 주지 않는 선에서 `asyncio`와 `aiohttp`를 도입하여 다중 객체를 동시에 크롤링하면 소요 시간을 비약적으로 단축할 수 있습니다.
* **컨테이너화(Dockerization)**: Playwright(브라우저) 의존성, 파이썬 패키지, MariaDB 설정을 일관되게 배포하고 구동할 수 있도록 `docker-compose` 환경으로 묶어주는 것이 장기적 유지보수에 유리합니다.

## 시작하기 (Getting Started)

1. 의존성 패키지를 설치합니다. Playwright 구동을 위한 브라우저 설치가 포함됩니다.
   ```bash
   pip install -r requirements.txt
   playwright install
   ```
2. 프로젝트 루트 경로에서 마스터 스크립트를 실행합니다.
   ```bash
   python all.py
   ```
3. 콘솔에 출력되는 메뉴 지시에 따라, 과거 데이터 수집, 출전표 수집, 실시간 감시 모드 중 하나를 선택하여 파이프라인을 가동합니다.
