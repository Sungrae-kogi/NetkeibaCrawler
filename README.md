# 🐎 Netkeiba Master Crawler Pipeline

## 📖 프로젝트 연혁 및 배경 (History)

### 1️⃣ 초기 단계 : 단일 `WebCrawler` 프로젝트
* **시작점**: 본 프로젝트는 원래 일본 경마 사이트 Netkeiba(넷케이바)에서 특정 경기의 출전표와 과거 경기 결과 데이터만을 수집하기 위한 가벼운 단일 크롤러(`WebCrawler`) 로 시작되었습니다.
* **주요 기능**: 사용자로부터 단일 URL을 입력받아 `Requests`와 `BeautifulSoup`을 통해 경기의 표면적 데이터만 파싱하여 CSV로 덤프하는 것이 핵심이었습니다.

### 2️⃣ 데이터의 조각화 및 고도화 필요성 대두
* **한계 직면**: 더욱 수준 높은 경마 데이터 분석을 위해서는 표면적인 경기 결과 뿐만 아니라, 출전하는 **말(Horse), 기수(Jockey), 조교사(Trainer)**의 누적 프로필과 디테일한 과거 성적(통계) 데이터가 독립적으로 필수적이었습니다. 
* 더불어 이미 끝난 경기뿐만 아니라, **다가올 미래의 출전 예정표(Entry Sheet / Shutuba)**를 분석하는 기능도 추가로 요구되었습니다.
* **파편화 발생**: 이를 해결하기 위해 `HRNOCrawler`, `JKNOCrawler`, `TRNOCrwaler` 등 각 객체의 프로필을 심층 파싱하는 독립적인 서브 크롤러들이 생겨났습니다. 하지만 각각을 일일이 실행하고, 메인 데이터에서 ID를 수동으로 복사해서 옮겨줘야 하는 심각한 병목 현상이 발생했습니다.

### 3️⃣ 해결책 : 통합 자동화 마스터 파이프라인 구축
위와 같은 병목과 유지보수의 한계를 타파하고자, 뿔뿔이 흩어진 모든 서브 크롤러들을 하나로 묶고 통제하는 **통합 오케스트레이션 프로그램 (`all.py`)** 체제로 전면 개편되었습니다.

## ⚙️ 현재 시스템 구조 및 핵심 로직 (Architecture)

현재 저장소는 다음과 같이 완전 자동화된 파이프라인으로 동작합니다.

### 👑 `all.py` (Master Orchestrator)
* 대화형(CLI) 메뉴를 통해 **모드 1(과거 경기 결과 수집)** 과 **모드 2(미래 출전 대상 수집)** 를 선택할 수 있는 컨트롤 타워입니다.
* 사용자는 단 한 번의 URL 입력만 하면 되며, 이후의 모든 데이터 추출 및 병렬 가동은 자동으로 이루어집니다.

### 🏗️ Phase 1 & 2 : Base 크롤링 및 ID 분배 (`WebCrawler`)
* 입력받은 URL을 토대로 베이스가 되는 경기 정보 메인 CSV 파일을 먼저 생성합니다.
* 생성된 마스터 데이터에서 유니크한 `HRNO`(말 고유번호), `JKNO`(기수 고유번호), `TRNO`(조교사 고유번호)를 모두 빼내어 각각의 서브 크롤러의 `nodata/` 대기열에 날짜별로 분배합니다. 

### ⚙️ Phase 3 : Sub 크롤러 연쇄 가동 (`HR/JK/TR Crawler`)
* Base 단계가 끝나면, `all.py`가 곧바로 3개의 프로필 크롤러를 순차적으로 런칭합니다.
* 각 크롤러는 대기열에 들어온 ID 리스트를 읽어들여 개별 페이지에 접속, 디테일 정보를 크롤링한 뒤 `data/` 경로에 최종 결과물을 뱉어냅니다.
* 이미 수집했던 데이터는 중복해서 긁지 않도록 자체 캐싱 필터링이 구현되어 있습니다.

## 🚀 사용 가이드 (How to Use)
1. 프로젝트 루트 경로에서 아래 명렁어를 실행합니다.
   ```bash
   python all.py
   ```
2. 콘솔에서 원하는 수집 모드(1번 시합 결과, 2번 엔트리 시트)를 선택합니다.
3. 넷케이바에서 복사해 온 URL을 붙여넣으면 나머지 모든 프로필 분배 및 수집 과정이 전자동으로 처리됩니다.
