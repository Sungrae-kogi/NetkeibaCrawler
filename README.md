# 🐎 Netkeiba Master Crawler Pipeline

## 📖 프로젝트 연혁 및 배경 (History)

### 1️⃣ 초기 단계 : 단일 `WebCrawler` 프로젝트
* **시작점**: 본 프로젝트는 원래 일본 경마 사이트 Netkeiba(넷케이바)에서 특정 경기의 출전표와 과거 경기 결과 데이터만을 수집하기 위한 가벼운 단일 크롤러(`WebCrawler`) 로 시작되었습니다.
* **주요 기능**: 사용자로부터 단일 URL을 입력받아 `Requests`와 `BeautifulSoup`을 통해 경기의 표면적 데이터만 파싱하여 CSV로 덤프하는 것이 핵심이었습니다.

### 2️⃣ 데이터의 조각화 및 고도화 필요성 대두
* **한계 직면**: 더욱 수준 높은 경마 데이터 분석을 위해서는 표면적인 경기 결과 뿐만 아니라, 출전하는 **말(Horse), 기수(Jockey), 조교사(Trainer)**의 누적 프로필과 디테일한 과거 성적 데이터가 독립적으로 필수적이었습니다. 
* 더불어 이미 끝난 경기뿐만 아니라, **다가올 미래의 출전 예정표(Entry Sheet / Shutuba)**를 분석하는 기능도 추가로 요구되었습니다.
* **파편화 발생**: 이를 해결하기 위해 `HRNOCrawler`, `JKNOCrawler`, `TRNOCrwaler` 등 각 객체의 프로필을 심층 파싱하는 독립적인 서브 크롤러들이 생겨났습니다. 하지만 각각을 일일이 실행하고, 메인 데이터에서 ID를 수동으로 복사해서 옮겨줘야 하는 심각한 병목 현상이 발생했습니다.

### 3️⃣ 해결책 : 통합 제어 및 실시간 모니터링 체제 구축
위와 같은 병목과 유지보수의 한계를 타파하고자, 뿔뿔이 흩어진 모든 서브 크롤러들을 하나로 묶고 제어하는 **통합 오케스트레이션 메인 시스템 (`all.py`)** 으로 전면 개편되었습니다.  
나아가, 주말 경마 기간 중 실시간으로 변동되는 데이터(출주 취소, 기수 변경 등)를 감지하는 **모니터링 봇(`InformationCrawler`)** 이 파이프라인에 추가 연동되었으며, 파이프라인 전반의 입출력 폴더(`data`, `nodata`) 구조가 통일되고 Git 추적 환경이 정리되었습니다.

## ⚙️ 현재 시스템 구조 및 핵심 로직 (Architecture)

현재 저장소는 다음과 같이 완전 자동화된 파이프라인 구조를 갖추고 있습니다.

### 👑 `all.py` (Master Orchestrator)
대화형(CLI) 메뉴를 통해 3가지 모드로 가동할 수 있는 핵심 컨트롤 타워입니다. 사용자가 단 1번의 액션만 취하면 모든 데이터 추출과 병렬 작업이 자동으로 진행됩니다.

* **모드 1**: 과거 경기 결과 수집 모드 (result)
* **모드 2**: 미래 경기 계획/출전표 수집 모드 (shutuba/entry_sheet)
* **모드 3**: 실시간 변경 정보 모니터링 모드 (InformationCrawler)

### 🏗️ Phase 1 & 2 : Base 크롤링 및 ID 분배 (`WebCrawler`)
* 입력받은 넷케이바 경기 URL을 토대로 메인 데이터 파일(`race_planning_*.csv`)을 `WebCrawler/data/` 에 1차 생성합니다.
* 마스터 데이터에서 식별 가능한 `HRNO`(말), `JKNO`(기수), `TRNO`(조교사) 고유번호를 모두 추출하여 각 서브 크롤러의 `nodata/` 경로에 자동 분배(Dispatch)합니다.

### ⚙️ Phase 3 : Sub 크롤러 연쇄 가동 (`HR/JK/TR Crawler`)
* 분배가 완료되면 `all.py`가 3개의 프로필 크롤러를 순차적으로 런칭합니다.
* 대기열에 들어온 ID 리스트를 읽어들여 상세 페이지를 크롤링하고, 첫 번째 컬럼으로 경기장(`MEET`) 정보를 자동 기록하며 최종 결과를 각 프로젝트의 `data/` 경로에 저장합니다. 
* 넷케이바의 EUC-JP 인코딩 환경에 완전히 최적화되어 데이터 깨짐 현상을 방지하며, 자체 해시 및 캐싱 로직으로 중복 수집 부하를 막습니다.

### 📡 실시간 모니터링 (`InformationCrawler` - 모드 3)
* 넷케이바 홈페이지의 Information 탭을 일정 간격(1시간)마다 읽어와 신규 공지가 등록되었는지 추적합니다. (대기 중 언제든 `q` 키 입력으로 메인 메뉴로 안전하게 복귀 가능)
* **출주 취소 (出走取消)** 건이 확인될 경우, 내부 정규화를 거쳐 경기장, 레이스 숫자, 마번, 말 이름 등을 세밀하게 발췌해 `cancel_extracted_info.csv` 파일로 모델링하여 자동 분리 저장합니다. 원본 공지는 `extracted_info.csv`로 단일 로깅됩니다.

## 🚀 사용 가이드 (How to Use)
1. 프로젝트 루트 경로에서 아래 명령어를 실행합니다.
   ```bash
   python all.py
   ```
2. 콘솔에 출력되는 전체 메뉴에서 원하는 모드(1, 2, 3)를 선택합니다.
3. **1, 2번 모드**: 넷케이바에서 복사해 온 URL을 붙여넣으면 나머지 프로필 분배 및 수집 과정이 관리자 개입 없이 수행됩니다.
4. **3번 모드**: 주말 등 실시간 감시가 필요할 때 켜두면, 백그라운드 프로세스처럼 1시간 간격으로 변경사항을 추적하고 알립니다.
