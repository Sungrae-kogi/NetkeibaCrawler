# 자동 복구(Auto-Healing) 크롤링 파이프라인 기획 

이 문서는 단순 1회성 크롤링 스크립트를 넘어, **"대형화/자동화/실시간 대응"**이 가능한 강력한 데이터 수집 시스템을 만들기 위해 사용자와 AI가 논의하고 확립한 구조적 방향성에 대한 기록입니다.

## 1. 프로젝트의 궁극적 비전 (사용자의 목표)
- **다양한 타겟 및 주기**: 여러 종류의 페이지(결과, 계획, 기수, 말, 조교사 프로필 등)를 각각 다른 주기로, 혹은 실시간 데이터가 유입되는 특정 시간 구간 동안 지속적으로 긁어와야 함.
- **연속 모니터링**: 한 번 돌고 끝나는 게 아니라, 켜두면 부족한 부분을 알아서 채우도록 동작하는 백그라운드 시스템 지향.
- **강건성(Robustness)**: 네트워크 오류, 서버 상태 불량 등으로 크롤링이 중간에 실패하더라도 프로그램이 "뻗어버리지(Crash) 않고", 스스로를 회복(재시도)해야 함.

## 2. 해결해야 했던 핵심 과제
1. 프로그램 강제 종료 방지 (Crash Prevention)
2. 실패 시 똑똑한 재시도 (Smart Retry Limit)
3. **데이터 무결성 (중복 X, 누락 채우기 O)**: 가장 중요한 부분. 다시 돌렸을 때 같은 데이터를 두 번 쓰지 않고 구멍만 정밀 타격해서 수집해야 함.

## 3. 합의된 해결 구조 (Auto-Healing Loop)
저희는 로그를 파싱하는 별도의 무거운 스크립트를 짜는 대신, 파이프라인 자체를 "파일(CSV) 중심"으로 설계하여 단순하고 완벽한 로직을 택했습니다.

### 3.1 하위 크롤러 레벨 (Self-Filtering 기능)
1. **중복 검증 로직 탑재**: (`HRNO`, `JKNO`, `TRNO` 모두) 자신이 생성한 결과물 `CSV` 파일을 스스로 읽어 검사하는 기능(`get_completed_list()`)을 갖춥니다.
2. 수집해야 할 전체 명단을 받으면 이 중 **"이미 CSV에 성공적으로 쓰여진" 녀석들은 바로 스킵**합니다.
3. 오직 "한 번도 성공한 적 없는(누락된)" 타겟만 골라서 찌릅니다.
4. 크롤링 자체의 1차 방어는 `HTTPAdapter(max_retries=3)`를 통해 순간적인 네트워크 단절을 버티도록 설계되어 있습니다.
5. 그럼에도 불구하고 실패하는 악성 타겟이 있다면 에러 로그만 남기고 일단 다음 녀석으로 넘어갑니다 ("부분 실패").
6. 종료 시, 만약 실패한 항목이 1개라도 있었다면 시스템 에러 코드(`sys.exit(2)`)를 반환하여 마스터에게 보고합니다.

### 3.2 마스터 프로세스 레벨 (`all.py` Orchestrator)
1. `HRNO -> JKNO -> TRNO` 순으로 1회차 사이클을 돌립니다. (`subprocess.run`)
2. 돌리던 중 스크립트 자체가 터지면(Crash) 완전히 멈추고 `[CRITICAL]` 로그를 남기지만, "부분 실패(`sys.exit(2)`)"로 종료되었다면 마스터는 이를 기억합니다.
3. 전체 3개 크롤러가 사이클을 다 돌고 난 후, 마스터가 진단합니다: "누락된 항목이 있는가?"
4. 누락이 발견되면, **최대 3회까지 해당 사이클을 재가동(Loop)** 합니다.
5. 2회차 사이클 진입 시, 하위 크롤러는 자기가 방금 전 1회차 때 남겼던 CSV를 읽고 시작하므로(3.1번 항목), **자연스럽게 중복은 패스하고 누락된 녀석들만 정밀 재수집**하게 되는 마법 같은 자동 복구가 이루어집니다.
